## 🔥 关键改动:namespace MCP 工具调用全栈修复 + MiMo 多模态纯图兜底

v2.1.1 主体是**根治 namespace 包装的 MCP 工具调用**。用户配 `~/.codex/config.toml` 多 MCP server 时,Codex.app 实际把每个 server 的工具集打成 `type:"namespace"` 包装,代理之前全部 drop,模型只能用 `read_mcp_resource` 等通用 meta function 间接读资源,**无法调用具体 MCP tool**(如 `notion_search` / `figma_get_metadata` / `zeabur_*` 等)。本版五层修复后,Notion / Figma / Zeabur 等 MCP server 完整工具集模型可见可调。

实地 dump 实测:用户配 12 MCP server 时,每轮入站含 218 个 namespace × 88 个内层 function tool,P5 之前全部丢失,P5 之后全部 work。

### 1. namespace MCP 工具调用全栈修复

#### 出站(让上游模型能看见具体 MCP tool)

`convert_responses_tool_to_chat_tool` 函数签名 `Option<Value>` → `Vec<Value>`,新增 `"namespace"` 分支递归 `flat_map` 展平内层 functions(借鉴 [`7as0nch/mimo2codex`](https://github.com/7as0nch/mimo2codex) `reqToChat.ts:232-250` 同名 namespace flatten 逻辑)。空 / 缺失 tools 字段静默 drop;内层 function 不改 name(Codex.app 已自带 server 前缀如 `notion_search`)。同时新增 `warn_once_drop_tool` helper(`crates/adapters/src/lib.rs`)防多轮重发 unknown type 工具时 log 刷屏。

#### 入站(让 Codex.app 能 dispatch namespace 工具调用)

`RequestPlan` 加 `original_responses_request: Option<Value>` 字段,在 `prepare_request` 阶段保存入站完整 Responses API request body。`ChatToResponsesConverter` 重构:

- envelope 回灌入站 16+ 字段(`tools` / `tool_choice` / `parallel_tool_calls` / `reasoning` / `text` / `metadata` / `previous_response_id` / `instructions` / `temperature` / `top_p` / `max_output_tokens` / `truncation` / `created_at`),严格 Responses API 协议合规(借鉴 mimo2codex `streamToSse.ts:75-105` `buildResponseSnapshot`)
- 每个 SSE event 自带单调递增 `sequence_number`(`streamToSse.ts:71-72`)
- **核心修复**:`function_call` SSE output 加 `namespace` 字段。通过 `strings /Applications/Codex.app/Contents/Resources/codex` 实证 Codex.app 桌面 binary 含 `"dynamic tool namespace must not be empty for"` 校验和 `DynamicToolCallRequest` 6 字段反序列化 struct,**缺这个字段时所有 namespace 包装的 MCP 工具 dispatch 都返回 `unsupported call: <name>`**。代理在 `with_original_request` 注入时扫 tools 数组建立 `function_name → namespace_name` 反查表,emit `function_call` 相关 events 时给 item 加 `namespace` 字段

#### 调研路径教训

第一轮 dump 抓 `plan.body`(prepare_request **之后**的 outbound chat body),namespace 已被 drop 看到 0 个 namespace,误判为"Codex CLI 不发 namespace"。用户质疑"GPT 自己怎么调?"才意识到要抓**入站** body_bytes 对照,重 dump 看到真相 — 入站每轮 218 个 namespace 包装。教训:dump 位置错一行,结论可以完全反向。

### 2. MiMo 多模态纯图请求兜底

`ensure_text_part_when_image_present` 兜底:扫每个 message,若 content 是 array 且含 `type:image_url` 但不含 `type:text` block,末尾追加 `{type:"text", text:" "}`(单空格)。1:1 复刻 mimo2codex `reqToChat.ts:71-79`。

**触发场景**:MiMo 多模态[官方文档](https://platform.xiaomimimo.com/docs/zh-CN/usage-guide/multimodal-understanding/image-understanding)强制要求 content 含 `image_url` 时至少 1 个 text part,否则上游 400 `Param Incorrect: text is not set`。Codex CLI 直接粘图未输入文字时 user 消息 content 数组只有 image_url 一个 block,触发该错误。

**范围**:统一对所有 `supports_vision` provider 应用(MiMo 必需,Kimi / OpenAI 等无副作用),避免 per-provider 判断。

## English

This release fixes namespace-wrapped MCP tool call dispatch end-to-end. Codex App's runtime sends each `[mcp_servers.<name>]` config entry as a `type:"namespace"` tool pack with ~10-30 inner functions; previous releases dropped these entirely on the proxy side, leaving the model unable to invoke specific MCP tools — only the generic `read_mcp_resource` meta functions worked, returning `unsupported call: <name>` for direct calls.

Real dump shows: with 12 MCP servers configured, each turn contains 218 namespace × 88 inner function tools, all lost before P5, all work after P5.

### 1. namespace MCP tool dispatch — five-layer fix

**Outbound** (so upstream model sees specific MCP tools):
- `convert_responses_tool_to_chat_tool` recursively `flat_map`s `type:"namespace"` packs into top-level function tools (borrows `7as0nch/mimo2codex` `reqToChat.ts:232-250`)
- `warn_once_drop_tool` helper prevents per-turn re-warn spam for unsupported types

**Inbound** (so Codex App can dispatch namespace tool calls back to MCP servers):
- `chat→responses` SSE envelope replays all 16+ Responses API fields (`tools`/`tool_choice`/`reasoning`/`text`/`metadata`/`previous_response_id`/`temperature` etc.) + `created_at` + monotonically increasing `sequence_number` for strict-protocol compliance
- **Critical fix**: adds `namespace` field to `function_call` SSE output — `strings` against `/Applications/Codex.app/Contents/Resources/codex` binary surfaces `"dynamic tool namespace must not be empty for"` validation and `DynamicToolCallRequest` 6-field deserialization struct. Without this field, every namespace-bound tool call fails with `unsupported call: <name>`. The proxy builds a `function_name → namespace_name` lookup table at `with_original_request` time and attaches `namespace` to function_call items in `response.output_item.added` / `.done` and `response.completed.output[]`

### 2. MiMo multimodal image-only fallback

MiMo's vision API requires at least one `text` part in the content array when `image_url` parts exist, otherwise 400 `Param Incorrect: text is not set`. Covers the Codex CLI paste-image-without-typing case by appending a single-space text part. Applied uniformly to all `supports_vision` providers (no side-effect on Kimi/OpenAI etc.).

### 3. Compliance improvements

- macOS Intel x64 build matrix fix (closes #61): `macos-13` runner retired 2025-12-04, GitHub silently dropping jobs for 24h+; switched to `macos-15-intel` (free for public repos, supported through 2027-08). Intel users can download `Codex-App-Transfer-v2.1.1-macOS-x64.dmg` directly
- README credits: `7as0nch/mimo2codex` for namespace flatten + image text part reference implementations
