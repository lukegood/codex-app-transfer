## 🔥 关键修复:Codex CLI 长会话 / ws 路径首条 400 + 续轮卡顿

**症状**:打开 app 后 Codex CLI 新会话**第一条对话必触发 19 秒 RTT 后 400**(Kimi Code 实测),日志大量 `messages must not be empty`;长会话续轮也偶发 400 后"卡几分钟"才恢复。

**根因(三方对账修正)**:不是早期假设的"cache miss"。Codex CLI `prewarm_websocket`(`core/src/client.rs:1334-1343`)启动时发 ws frame `{input: [], generate: false}` —— OpenAI Responses ws 协议预热语义,服务端应当只建连不生成。我们代理 `server.rs` 不识别 `generate: false`,**盲目把 frame 转 HTTP** 给上游,转换后 `messages: []`,任何 chat-completions 兼容 provider 必拒。"中途 400 后能继续"也不是 Codex 重试 400(`is_retryable` 对 `InvalidRequest` 永远 false,turn loop 直接 fail-fast);而是代理把 400 包成 ws stream error → Codex 当 `CodexErr::Stream` retryable → 5 次 stream retry 失败 → `try_switch_fallback_transport` 切 HTTP → HTTP 路径发完整 history 自然成功。"几分钟"是 ws retry 浪费 + HTTP 重 turn 累计。

**修法(三段)**:

(1) **WS handler 识别 warmup / 空 frame 不转 HTTP + 用 ws Close frame 让客户端立即 fallback**(PR #67 + 修正 PR #70):**注**:PR #67 早期版本用 `{"type":"error",...}` 文本帧被 Codex CLI 当流内错误处理,等 ws idle timeout ~5 分钟才 fallback(实测反馈 fb-8f5b51fb / fb-0c121681 卡 4 分 48 秒),PR #70 改用 `Message::Close` 协议层连接关闭信号,Codex CLI 立即 fallback。`server.rs` 收 ws frame 后,`generate: false` 或空 `input` + 无 `previous_response_id` → 直接 `send_ws_error` 返回 `type: error`,**不打上游**。每次首条 19s+ 上游 RTT 归零。**保留** `input:[] + previous_response_id != ""` 走转发(ws incremental delta=0 续轮命中 cache)。

(2) **Responses 会话历史持久化到 SQLite**(PR #65):`ResponseSessionCache` 双层 — L1 `Mutex<HashMap>` 1000×60min LRU 保留热路径零 IO;L2 SQLite `~/.codex-app-transfer/sessions.db` 30 天 TTL,WAL + `synchronous=NORMAL`,write-through 双写。Tauri 重启 / 长时间空闲后 ws incremental delta=0 续轮命中 L2 → 拼历史 + 空 delta → ws 直通成功,避开 stream retry + HTTP fallback 浪费循环。schema mismatch 自动备份重建,IO 失败回退纯内存。新 admin endpoint `POST /api/sessions/clear` 支持隐私清除。

(3) **cache miss 错误体对齐 OpenAI SDK**(PR #64):cache miss + empty input 在代理本地立即返回 OpenAI SDK 兼容 HTTP 400 + `{"error":{"code":"previous_response_not_found",...}}`,字面对齐 OpenAI Responses API 服务端真实行为(LM Studio bug tracker #1188、Microsoft semantic-kernel #13128 多源验证)。延迟从分钟级降到几十毫秒。

|---|---|---|
| 新会话首条 prompt | ❌ 19s+ Kimi RTT 后 400 → ws retry 5 次(~6s) → fallback HTTP | ✅ warmup 不打上游 → ~ms 错误 → fallback HTTP(残留 ~6s ws retry 客户端无法消除) |

## 🔥 Critical fix: Codex CLI long-session / ws first-turn 400 + follow-up stalls

**Symptoms**: every fresh Codex CLI session's first prompt **always triggered a 19-second RTT then 400** (Kimi Code measured), with `messages must not be empty` flooding the log; long-session follow-ups also showed sporadic mid-conversation 400 stalls of "several minutes" before recovery.

**Root cause (corrected after a thorough audit)**: Not "cache miss". Codex CLI's `prewarm_websocket` (`core/src/client.rs:1334-1343`) opens every session by sending a ws frame `{input: [], generate: false}` — the OpenAI Responses ws-protocol "warmup" semantics: open the connection, do not generate. Our proxy `server.rs` did not recognize `generate: false` and **blindly forwarded the frame as an HTTP POST** to the upstream, where it became `messages: []` — rejected by every chat-completions-compatible provider. "Mid-session 400 then recovery" is also not Codex retrying 400 (`is_retryable` returns `false` for `InvalidRequest`, turn loop fails fast); rather, the proxy wrapped the 400 into a ws stream error event, which Codex treats as `CodexErr::Stream` (retryable), runs 5 stream retries, and on exhaustion calls `try_switch_fallback_transport` to fall back to HTTP. The HTTP path sends full history and naturally succeeds; the "minutes" are wasted ws retries + the rerun of the full HTTP turn.

**Fix (three parts)**:

(1) **WS handler skips warmup / empty frames + sends ws Close frame so the client falls back immediately** (PR #67 + corrected by PR #70): **Note**: an earlier PR #67 build sent `{"type":"error",...}` text frames, which Codex CLI treated as a stream-level error and waited the full ws idle timeout (~5 min) before falling back (feedback fb-8f5b51fb / fb-0c121681 measured 4m48s); PR #70 switched to `Message::Close` (protocol-level connection-close signal), so Codex CLI falls back immediately. `server.rs` inspects each frame after extracting the body. If `generate: false` or `input` empty AND no `previous_response_id`, the proxy directly sends a ws `type: error` and **does not touch the upstream**. The 19s+ upstream RTT on every session's first turn is gone. **Preserved**: `input: [] + previous_response_id != ""` (ws incremental delta=0 follow-up) still goes through the cache lookup path.

(2) **Persistent Responses session cache** (PR #65): two-tier `ResponseSessionCache` — L1 `Mutex<HashMap>` 1000×60min LRU keeps the hot path zero-IO; L2 SQLite at `~/.codex-app-transfer/sessions.db` with 30-day TTL, WAL + `synchronous=NORMAL`, write-through both layers. Tauri restarts and long idle no longer wipe history; ws incremental delta=0 follow-ups hit L2 → assemble history + empty delta → upstream → ws path streams through, **avoiding the stream-retry + HTTP-fallback waste cycle**. Schema mismatch auto-backs-up and rebuilds; IO failure falls back to in-memory only. New admin endpoint `POST /api/sessions/clear` for privacy control.

(3) **Cache-miss error body aligned with OpenAI SDK** (PR #64): cache miss + empty input returns an OpenAI SDK-compatible HTTP 400 + `{"error":{"code":"previous_response_not_found",...}}` directly from the proxy, literally matching OpenAI Responses-API server-side behavior (verified against LM Studio bug tracker #1188, Microsoft semantic-kernel #13128). Latency drops from minutes to tens of milliseconds.

| Codex CLI scenario | v2.0.10 | v2.0.11 |
|---|---|---|
| Fresh-session first prompt | ❌ 19s+ Kimi RTT then 400 → 5× ws retry (~6s) → HTTP fallback | ✅ warmup not forwarded → ~ms error → HTTP fallback (residual ~6s ws retry can't be eliminated client-side) |
| Long-session ws follow-up (delta=0) | ❌ in-process cache wiped on restart → always HTTP fallback | ✅ L2 SQLite 30-day hit → ws streams through |
| Residual cache miss / edge cases | ❌ 19s upstream 400 + confusing error | ✅ ms-level proxy-local SDK-compatible 400 + standard error.code |
