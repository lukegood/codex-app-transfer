# Codex App Transfer v2.1.9

## 中文

### Codex CLI 0.126+ 兼容修复与第三方模型工具调用稳定性

本版本聚焦两类问题:Codex CLI 0.126+ 端点变更后 openai_chat 路径暴露的 `404 /v1/responses` 问题(issue #178),以及 DeepSeek / Kimi 等严格校验上游在长会话并行工具调用场景下的 400 问题(issue #180,会话一旦"中毒"无法继续)。

主要改动:

- apply 阶段强制写 `model_provider = "openai"`,避免用户旧 config 残留的 `model_provider = "custom"` + `[model_providers.custom]` 段把流量旁路到第三方 base_url。Codex CLI 0.126+ 把端点从 `/v1/responses` 切到 `/responses` 后,残留路径直接表现为 404。
- 快照在 apply 第一步已拿到用户原值,stop app 时 restore 能完整退回到原 `model_provider`(或缺省状态),用户的 `[model_providers.*]` 段不受影响。
- restore 副作用修复:用户在 Codex CLI 模型选择器选过的 `model` 在 stop app 时不再被擦掉,旧 session 续接不再切回内置 GPT 名。手动选 snapshot 恢复路径仍严格按快照恢复,两类语义分开。
- `repair_tool_call_ids` 加反向修复:assistant.tool_calls 有 `[a, b, c]` 但只跟了 tool b 时,在 user/system/developer 边界 + 末尾 + 新 assistant 前注入占位 tool 消息,id 严格匹配,避免严格校验上游 400。占位 content 沿用 litellm `_add_missing_tool_results` 措辞,tool name 从 `ToolCallCache` 反查,miss 退化为 `unknown_tool`。

## English

### Codex CLI 0.126+ compatibility fix and third-party model tool-call stability

Two classes of issues are fixed: the `404 /v1/responses` exposed by the Codex CLI 0.126+ endpoint change on the openai_chat path (issue #178), and the long-session parallel tool-call 400 reported against strict upstreams like DeepSeek / Kimi (issue #180, where a poisoned session previously could not recover).

Changes:

- Apply now forces `model_provider = "openai"` so user configs carrying a stale `model_provider = "custom"` + `[model_providers.custom]` block can no longer route traffic past the proxy. Codex CLI 0.126+ switched its endpoint from `/v1/responses` to `/responses`, so the stale path surfaces as a 404.
- Snapshot captures the user's original value before apply, so stop-app restore reverts `model_provider` cleanly and leaves `[model_providers.*]` blocks untouched.
- Restore side-effect fix: a `model` picked via the Codex CLI selector is preserved at stop-app restore, so old sessions no longer flip back to a built-in GPT name. Manual snapshot restore still mirrors the chosen snapshot strictly — the two paths are split.
- `repair_tool_call_ids` now also handles the reverse case: when assistant.tool_calls has `[a, b, c]` but only tool b is answered, placeholder tool messages are injected at user/system/developer boundaries, at end of input, and before a new assistant — ids matched strictly to avoid the strict-upstream 400. Placeholder content follows litellm `_add_missing_tool_results`; tool name is resolved from `ToolCallCache` with `unknown_tool` as fallback.
