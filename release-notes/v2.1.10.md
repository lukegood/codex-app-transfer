## English

### Alibaba Cloud Bailian Token Plan end-to-end + Plugin Unlock auto-start stability

Bailian Token Plan goes from connection-failure to fully end-to-end working; Codex Desktop Plugins unlock now defaults on with daemon-status race fixed; autocompact threshold lifted so PreTurn nearly always wins to avoid mid-turn interruption; new root `ACKNOWLEDGEMENTS.md` centralizes upstream-borrowing records.

Changes:

- Bailian Token Plan preset `apiFormat` changed from `responses` to `openai_chat`, fixing connection failure (`/compatible-mode/v1` only supports Chat Completions; Responses passthrough returns 404).
- For Bailian Token Plan host (`token-plan.cn-beijing.maas.aliyuncs.com`), `fetch_provider_models_impl` short-circuits to hardcoded 4 models (`qwen3.6-plus`/`deepseek-v3.2`/`glm-5`/`MiniMax-M2.5`), borrowed from Alibaba's official `QwenLM/qwen-code` `TOKEN_PLAN_MODELS` (Apache-2.0); the upstream gateway gates auth before routing, returning 401 for any unknown path so probing is impossible.
- Plugin Unlock daemon `autoUnlockCodexPlugins` now defaults to true, shares an OnceCell daemon singleton, removes silent fallback, fixing the `/api/plugin-unlock/status` `Disconnected` race.
- Autocompact `AUTO_COMPACT_TRIGGER_PERCENT` raised 75% → 80%, giving a 256K context ~51K buffer covering typical 5K-30K single-turn deltas so PreTurn nearly always wins.
- Removed the "verified end-to-end / compatibility adapter" banner from the provider-add form; new `ACKNOWLEDGEMENTS.md` upstream-borrowing catalog (9-field schema with License / borrowing form / this-project file:line / sync strategy).
