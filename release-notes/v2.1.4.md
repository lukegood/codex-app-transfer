## 🔥 关键改动:Gemini Native 直转适配器 + 错误流 SSE failure + 多轮 round-trip 修复

v2.1.4 主线是**新增 `apiFormat=gemini_native` 适配器**(Codex.app `/responses` → Google `:streamGenerateContent` 直转,完全不走 chat 中间形态),配套 11 项破坏性降级根治 + 错误流 → Responses SSE failure 转换 + 多轮 function calling round-trip 修复 + 清 9 条 builtin preset 27 行 dead legacy key。

### 1. Gemini Native 直转适配器(主线)

新增 `apiFormat=gemini_native` + `authScheme=google_api_key`(`x-goog-api-key` header)。Codex.app 入站 `/responses` **直接构造** Google `:streamGenerateContent?alt=sse` 出站,SSE 状态机直接 emit `response.created/in_progress/output_item.added/output_text.delta/function_call_arguments.delta/output_text.annotation.added/completed`,**不走 chat 中间**(`crates/adapters/src/gemini_native/`)。Gemini 3 → `/v1alpha`,Gemini 2.x → `/v1beta`,baseUrl 不带版本前缀适配器自动选。

核心兼容点(全部**非破坏性**,memory 用户硬性规则:不主动进行任何破坏性降级):

- **JSON Schema 兼容化**:`type:["string","null"]` 数组 → `string` + `nullable:true`;`anyOf` 含 null → 折叠 nullable;`$ref`/`$defs` 内联展开(LiteLLM unpack_defs 模式);`additionalProperties` / `strict` / 空 enum / 缺 properties 全部规范化
- **Web Search**:Gemini 3+ `googleSearch` builtin + functionDeclarations 共存(`toolConfig.includeServerSideToolInvocations`);Gemini 2.x 不允许共存 → `systemInstruction` 软约束告知模型(LiteLLM `apply_response_schema_transformation` 风格,**绝不 destructive drop 字段**);文案精确防 model 误把"联网"泛化拒所有 HTTP 操作(用户测试时模型仍能用 `exec_command` 跑 `curl`/`wget`)
- **多轮 function calling round-trip**:接全局 `ToolCallCache`(`crates/adapters/src/responses/tool_call_cache.rs`),`function_call_output` 缺前置 `function_call` 时按 `call_id` synthesize 假 assistant turn(根治 "function response without prior call" 400);`tool_choice` 按值 normalize(`auto/none` drop / `required/specific` BadRequest);`contents` 必须 user role 开头(strict alternation),首项是 model 时插占位 user turn

### 2. 错误流 → Responses SSE failure 流(修 Codex.app 卡 Thinking)

上游 4xx/5xx **不再透传 raw Gemini JSON** — 旧实现让 Codex.app 收到非 SSE raw JSON 不知怎么 parse → 永远不进 terminal 态(silent failure)。改成构造合规 SSE failure 流 `response.created → response.in_progress → response.failed`,`envelope.error` 用 OpenAI 标准 `{code, message, type}` 形态,客户端能识别 + 显示用户级错误。

错误分类按 status + Gemini `error.status` + message 关键词:`auth_error` / `permission_denied`(403 PERMISSION_DENIED 区分 401)/ `quota_exceeded`(429 RESOURCE_EXHAUSTED) vs `rate_limited`(429 其他)/ `timeout`(408/504)/ `service_unavailable`(503)/ `content_filter`(400 + safety) vs `bad_request`(400 其他)/ `server_error`(5xx 其他)/ `upstream_transport_error` / `upstream_error`。

防御性失败处理(经 silent-failure-hunter + code-reviewer 双 agent review,critical/high 项全修):ByteStream Err mid-read 捕获 transport err 拼进 message;body 64KB cap 防 DoS;非 UTF-8 / 2000 char 过长 message 标 marker 后缀;5xx + transport_err 走 `tracing::error!`,4xx 走 `tracing::warn!`;mod.rs strip `Content-Length` / `Content-Encoding`(避免下游 gunzip plaintext SSE → 整流 corrupt)。

### 3. 11 项破坏性降级根治 + 测试覆盖

用户复诊"function calling + responseMimeType 不能共存"修复时发现旧实现直接 drop 字段不告知(违反 "不主动进行任何破坏性降级" 硬规则),系统排查 29 候选点全修 P0×6 + P1×4 + P2×1。Gemini 单测 60+ 覆盖 wire round-trip / schema sanitize(array type / anyOf / strict / $ref)/ tool_choice modes / multi-turn cache / failure stream lifecycle / 错误分类 / array-form body / empty body / oversized body / non-UTF-8 / transport err。**290 adapter tests / 全 workspace test 全绿**。

### ⚠️ 使用注意:长对话切第三方 Responses provider 可能报 `thinking_signature_invalid`

Codex CLI 在长对话超出上下文阈值时会自动 compact 历史,塞 `type=compaction` item(Codex CLI **私有协议字段**,OpenAI Responses 标准协议**没有**;`encrypted_content` 字段名是历史包袱实际填明文 summary)。第三方 Responses 上游(OpenAI 官方 / Anthropic 系网关 / 自建 wrapper 等)不识别这个 item,会试图把 `encrypted_content` 当 thinking signature 去 verify → 报错。

**workaround**:切到第三方 Responses provider 时**新建对话**避免历史累积进 compact item;或者使用走代理转换的 builtin provider(`apiFormat=openai_chat`),代理本地 ResponsesAdapter 已正确把 `compaction` 展开成 user message 注入下游。

## English

### 1. Gemini Native direct adapter (main theme)

New `apiFormat=gemini_native` + `authScheme=google_api_key` (`x-goog-api-key` header). Codex.app `/responses` inbound directly emits Google `:streamGenerateContent?alt=sse` outbound — SSE state machine produces `response.created/in_progress/output_item.added/output_text.delta/function_call_arguments.delta/output_text.annotation.added/completed` events directly, **never via a chat intermediate** (`crates/adapters/src/gemini_native/`). Gemini 3 → `/v1alpha`, Gemini 2.x → `/v1beta`; baseUrl without version prefix is auto-resolved.

Compatibility points (all **non-destructive**, per the user's hard rule "no silent destructive fallback"):

- **JSON Schema compatibility**: `type:["string","null"]` → `string` + `nullable:true`; `anyOf` containing null → folded into nullable; `$ref` / `$defs` inlined (LiteLLM unpack_defs pattern); `additionalProperties` / `strict` / empty enum / missing properties all normalized
- **Web Search**: Gemini 3+ allows `googleSearch` builtin + `functionDeclarations` to coexist (`toolConfig.includeServerSideToolInvocations`); Gemini 2.x does not — falls back to `systemInstruction` soft constraint informing the model (LiteLLM `apply_response_schema_transformation` style, **never destructive-drops fields**); precise wording prevents over-generalizing to "no networking at all" (in user testing, the model still ran `curl`/`wget` via `exec_command`)
- **Multi-turn function calling round-trip**: hooks into the global `ToolCallCache`; when `function_call_output` is missing the prior `function_call`, looks up by `call_id` and **synthesizes the missing assistant turn** (fixes "function response without prior call" 400). `tool_choice` normalized by mode (`auto/none` drop, `required/specific` BadRequest). `contents` must start with `user` role (strict alternation); placeholder user turn inserted if first item is a `model` turn

### 2. Error stream → Responses SSE failure (fixes Codex.app stuck on "Thinking")

Upstream 4xx/5xx **no longer passes raw Gemini JSON through** — the old behavior left Codex.app stuck (it expects an SSE event stream, can't parse raw JSON, never reaches a terminal state — silent failure). Now synthesizes a compliant SSE failure stream `response.created → response.in_progress → response.failed`, with `envelope.error` shaped as the OpenAI standard `{code, message, type}` so clients identify and display a user-level error correctly.

Error classification by status + Gemini `error.status` + message keywords: `auth_error` / `permission_denied` (403 PERMISSION_DENIED, distinct from 401) / `quota_exceeded` (429 RESOURCE_EXHAUSTED) vs `rate_limited` (429 other) / `timeout` (408/504) / `service_unavailable` (503) / `content_filter` (400 + safety) vs `bad_request` (400 other) / `server_error` (5xx other) / `upstream_transport_error` / `upstream_error`.

Defensive failure handling (reviewed by silent-failure-hunter + code-reviewer agents, all critical/high findings fixed): mid-read transport errors are captured and inlined into the user message; body capped at 64KB to prevent DoS; non-UTF-8 / 2000 char overlong messages marked with suffixes; 5xx + transport_err emit `tracing::error!`, 4xx emit `tracing::warn!`; `Content-Length` / `Content-Encoding` stripped (otherwise the client tries to gunzip plaintext SSE → corrupt stream).

### 3. 11 destructive-fallback fixes + test coverage

After a user complaint about "responseMimeType silently dropped when functionDecls present" (violating the "no silent destructive fallback" hard rule), a systematic audit of 29 candidate sites fixed all P0×6 + P1×4 + P2×1. 60+ Gemini unit tests cover wire round-trip / schema sanitize / tool_choice modes / multi-turn cache / failure stream lifecycle / error classification / array-form body / empty body / oversized body / non-UTF-8 / transport err. **290 adapter tests / full workspace green**.

### ⚠️ Usage caveat: long conversations on third-party Responses providers may hit `thinking_signature_invalid`

When Codex CLI auto-compacts history beyond the context threshold it injects a `type=compaction` item (a **Codex CLI private-protocol field**, **not part of the OpenAI Responses standard**; the field name `encrypted_content` is a historical naming artifact — its value is actually plaintext summary). Third-party Responses upstreams (OpenAI official / Anthropic-style gateways / self-hosted wrappers) don't recognize this item and try to verify `encrypted_content` as a thinking signature → returns the error.

**Workaround**: when switching to a third-party Responses provider, **start a new conversation** to avoid history accumulating into a compaction item; or use a builtin proxy-routed provider (`apiFormat=openai_chat`) — the local ResponsesAdapter correctly expands `compaction` into user messages before forwarding.
