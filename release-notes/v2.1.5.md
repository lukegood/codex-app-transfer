## 🔥 关键改动:Gemini Native 软约束「minimal 默认 + 可配」+ Windows OAuth 修复 + Antigravity OAuth provider + 多实例并发安全收口

v2.1.5 主线是**修 Gemini Native 协议转换语义漂移**(默认改 `minimal` 软约束) + **加 Provider 表单 UI 让用户自选 `off/minimal/strict`** + **修 Windows 下 Gemini CLI OAuth provider 状态加载失败** + **新增 Google Antigravity OAuth provider 后端** + **跨进程 config 写并发收口**(file lock + 原子 RMW)。

### 1. Gemini Native 软约束默认 `minimal`,新增 Provider 表单 UI 自选

v2.1.4 实测发现:`gemini_native` 协议转换为了规避 Gemini 上游"functionDeclarations 不能与 responseMimeType/googleSearch 共存"的协议级冲突,会在 `systemInstruction` 注入一段冗长**软约束**(strict 文案)。该文案过于强势,会污染模型对原始 prompt 的语义理解,**模型回答偏离提示词**。

修复:

- **默认改 `minimal`**:`systemInstruction` 改为极简文案,只点明任务意图,不重复 wire-level 已经做的字段降级解释。仍然保留必要的协议防 400 降级(`response_mime_type` / `google_search` 在 wire 上 drop),只是不再把"为什么 drop"塞进 prompt
- **新增 `off`**:完全不注入 systemInstruction(wire 端必要降级仍生效),适合纯文本/不用 schema 的场景
- **保留 `strict`**:保留旧版详细文案,供 troubleshooting / 兼容性极端场景
- **UI 接入**(PR #118):Provider 表单新增「Gemini 兼容软约束」下拉,仅在 `apiFormat=gemini_native` 时显示,持久化为 `provider.compatSoftConstraints`。无需手改 `~/.codex-app-transfer/config.json`

### 2. Windows 下 Gemini CLI OAuth provider 修复「HOME unavailable」

用户反馈:macOS 上 Gemini CLI provider 登录正常,Windows 安装后状态条飘黄色"状态加载失败,HOME unavailable, HOME 环境变量未设置,无法定位 token 持久化目录"。

根因:Windows GUI 进程通常**没有 `HOME` 环境变量**(只有 `USERPROFILE`),`crates/gemini_oauth/src/token.rs` 旧实现只读 `HOME` 直接 fail。workspace 其他模块(`codex_integration` 等)已有 `HOME → USERPROFILE` 回退但这一处遗漏。

修复(PR #115 + #116):

- `TokenStore::for_token_filename` 改走 `codex_app_transfer_registry::paths::resolve_home()`,该函数按 `HOME → USERPROFILE` 顺序 + `var_os` 健壮处理 + 空字符串视为未设置
- 顺手把 workspace 内 3 处独立的 home 解析逻辑(`registry` / `codex_integration` / `gemini_oauth`)统一收敛到 `registry::paths::resolve_home` 单一入口,消除将来漂移
- 错误文案改成跨平台版:`无法定位 token 持久化目录:HOME 与 USERPROFILE 环境变量都未设置`
- 配套 9 个单测覆盖 home 解析 + display 契约

### 3. 新增 Google Antigravity OAuth provider 后端(PR #113)

新增 `apiFormat=antigravity_oauth` provider。跟 `gemini_cli_oauth` 共用 Cloud Code Assist `cloudcode-pa.googleapis.com` 上游,但使用 **Google Antigravity IDE 独立的 OAuth client_id + 独立 token 文件**(`~/.codex-app-transfer/antigravity-oauth.json`),与 Gemini CLI 凭证互不污染。

后端就绪后续 PR 将补 UI 完整路径(本版 Provider 表单暂未默认露出此项 preset,但 backend route + 协议路由 + token 持久化 + 状态/login/logout handler 全部已落地,可手工添加测试)。

### 4. 跨进程并发写收口:file lock + 原子 with_config_write

旧实现允许多个 `.app` 实例同时跑(用户实测会同时打开两个),但 config 文件读改写没跨进程锁 → race condition 概率丢用户改动。

修复(PR #99 / #106 / #107 / #108 / #109):

- **跨进程 file lock**(PR #109):`with_config_write` 加 fs2 advisory lock,多 `.app` 实例对 `config.json` 写串行化
- **原子 RMW API 全栈迁移**(PR #99 + #106):删 11 处 raw `load + 改 + save` 调用点,统一改 `with_config_write(|cfg| { ... })` 闭包形式
- **save() 改 module-private**(PR #107):防 future contributor 绕过原子 API 写盘
- **同进程 reentrant 检测**(PR #108):闭包内嵌套 `with_config_write` 直接 panic,把 silent 死锁变 loud panic

### 5. Gemini CLI OAuth 收口:取消 + app exit + HTTP client 池化

PR #100 / #101 / #103 / #104 / #105 系列补齐 OAuth 整链路:

- **登录可取消 + 抢占语义**:`oneshot` → `watch::channel`,cancel 信号贯穿整 login pipeline(浏览器授权 → token 交换 → project provisioning);新登录请求直接抢占进行中的旧请求
- **App exit 钩子**:应用退出时等 in-flight OAuth task 真正退出,避免 token 文件半写状态
- **HTTP client 池化**:多次 OAuth 操作共享同一 `reqwest::Client`(OnceLock),省连接握手开销
- **cancel response 端到端测试**:区分 poison-recovery vs preemption 两种取消语义

### 6. i18n 启动单帧闪烁修复(PR #102 + #110)

`data-i18n-pending` 属性 + inline CSS 在 `i18n.apply()` 完成前隐藏所有待翻译节点,消除"先闪一下中文 fallback 再切英文"的体感问题(EN 用户首屏 UX 修复)。同时 `formatI18n → tFmt` 统一,启动时序消除空窗乱码。

### 7. 杂项

- PR #112: README 致谢段每条压缩到 ≤ 20 字符
- PR #98: 加 `.mailmap` 把 AI 工具(Claude / Copilot / Cursor / cursoragent)产生的 git author 合并到 `Cmochance`,让 GitHub Contributors 反映真实贡献

## English

### 1. Gemini Native soft constraint defaults to `minimal`, with new Provider-form UI

After v2.1.4, real-world testing showed that `gemini_native` injected a long **strict-style soft constraint** into `systemInstruction` to navigate Gemini's wire-level "functionDeclarations cannot coexist with responseMimeType / googleSearch" conflicts. That text was so directive that it **polluted the model's understanding of the original prompt**, causing answers to drift away from user intent.

Fixes:

- **Default switched to `minimal`**: `systemInstruction` now uses concise wording that only restates task intent — wire-level field drops (`response_mime_type` / `google_search`) still happen to prevent 400s, but the "why" is no longer injected into the prompt
- **New `off` mode**: nothing is injected at all (wire-level drops still apply); ideal for plain-text / no-schema use cases
- **`strict` mode retained**: keeps the verbose legacy wording for troubleshooting / extreme-compat scenarios
- **UI wired up** (PR #118): Provider form now exposes a "Gemini compatibility soft constraints" dropdown (visible only when `apiFormat=gemini_native`), persisted as `provider.compatSoftConstraints` — no more hand-editing `~/.codex-app-transfer/config.json`

### 2. Windows: "HOME unavailable" fix for Gemini CLI OAuth provider

User report: on macOS the Gemini CLI provider logged in fine, but on Windows the status bar showed a yellow warning "状态加载失败, HOME unavailable, HOME 环境变量未设置, 无法定位 token 持久化目录".

Root cause: Windows GUI processes typically have **no `HOME` env var** (only `USERPROFILE`), and the old `crates/gemini_oauth/src/token.rs` only read `HOME` and fatally failed. Other workspace modules already had `HOME → USERPROFILE` fallback, but this one site was missed.

Fix (PR #115 + #116):

- `TokenStore::for_token_filename` delegates to `codex_app_transfer_registry::paths::resolve_home()`, which checks `HOME → USERPROFILE` using `var_os` (robust for non-UTF8 paths, treats empty string as unset)
- Consolidated **all three independent home-resolution implementations** across the workspace (`registry` / `codex_integration` / `gemini_oauth`) into a single canonical entry point to prevent future drift
- Error message is now platform-agnostic: `无法定位 token 持久化目录: HOME 与 USERPROFILE 环境变量都未设置`
- Backed by 9 unit tests covering home resolution + display contract

### 3. New Google Antigravity OAuth provider backend (PR #113)

New `apiFormat=antigravity_oauth` provider. It shares the Cloud Code Assist `cloudcode-pa.googleapis.com` upstream with `gemini_cli_oauth`, but uses the **Google Antigravity IDE's own OAuth client_id and a separate token file** (`~/.codex-app-transfer/antigravity-oauth.json`) — no cross-contamination with Gemini CLI credentials.

Backend is fully wired (routes, protocol dispatch, token persistence, status/login/logout handlers); the dedicated preset UI lands in a follow-up release, but the provider is usable now via manual configuration.

### 4. Cross-process config-write safety: file lock + atomic `with_config_write`

The app supports concurrent `.app` instances (some users genuinely run two windows), but config read-modify-write previously had no cross-process lock — a race window risked silently dropping user edits.

Fixes (PR #99 / #106 / #107 / #108 / #109):

- **Cross-process file lock** (PR #109): `with_config_write` now takes an fs2 advisory lock so concurrent `.app` instances serialize their writes to `config.json`
- **Workspace-wide migration to atomic RMW** (PR #99 + #106): 11 raw `load + mutate + save` call sites rewritten as `with_config_write(|cfg| { ... })` closures
- **`save()` made module-private** (PR #107): prevents future contributors from bypassing the atomic API
- **Same-process reentrant detection** (PR #108): nested `with_config_write` panics loudly instead of silently deadlocking

### 5. Gemini CLI OAuth hardening: cancellation, app-exit, pooled HTTP client

PR #100 / #101 / #103 / #104 / #105 finish the OAuth pipeline:

- **Cancelable login with preemption semantics**: `oneshot` → `watch::channel`; cancel signal threads through the whole login pipeline (browser authorization → token exchange → project provisioning); a new login request preempts any in-flight one
- **App-exit hook**: the app now awaits in-flight OAuth tasks on exit to avoid half-written token files
- **Pooled HTTP client**: a single `reqwest::Client` (OnceLock) shared across OAuth operations saves handshake overhead
- **End-to-end cancel-response tests**: poison-recovery vs preemption semantics covered

### 6. i18n first-paint flash fix (PR #102 + #110)

A `data-i18n-pending` attribute + inline CSS hides `[data-i18n]` nodes until `i18n.apply()` finishes — no more "Chinese fallback flashes for a frame, then switches to English" on first paint for EN users. Plus `formatI18n → tFmt` unified, and the startup ordering window that produced empty-string glitches is gone.

- PR #112: README acknowledgments compacted to ≤ 20 chars per line
- PR #98: `.mailmap` added to merge AI tooling git authors (Claude / Copilot / Cursor / cursoragent) into `Cmochance` so the GitHub Contributors list reflects actual humans
