## 目标

把当前 Rust/Tauri 版本相对 v1.0.3 漏掉的功能按重要等级拆成可推进任务。后续实现时按本文档任务编号推进, 每完成一项都需要补充执行记录和验证结果, 避免再次出现前端入口保留但后端实际为空实现的情况。

## 执行约束

- P0/P1 缺口默认按 `v1.0.x` 旧版逻辑直接翻译到当前 Rust/Tauri 架构, 不自行构思新逻辑替代旧逻辑。
- 后续 P0/P1 实现必须先完整对照并转化 `v1.0.x` 旧代码; 除测试、错误透出、当前架构必要适配外, 不新增旧版不存在的产品逻辑。
- 如果旧逻辑依赖的运行条件在当前架构中不存在, 必须先记录差异和迁移边界, 再做等价翻译; 不能用新的产品语义绕开。

## 优先级定义

- `P0`: 用户可见且会造成误导、假成功、点击异常或数据风险的缺口。优先处理。
- `P1`: v1.0.3 已有的核心产品能力, 当前缺失会明显降低可用性, 但通常不会立即破坏主链路。
- `P2`: 协议兼容、边界场景和长期正确性问题。影响特定请求形态或高级用法。
- `P3`: 文档、口径和清理项。需要跟随实现同步修正, 防止交接和发布说明误导。

## P0: 先消除假成功和直接可见故障

**问题**: 当前路由注释声称 `/api/*` shape 严格对齐 v1.4, 但多个接口仍是 stub。用户看到的是操作成功, 实际没有完成动作。

**证据**:

- 当前路由注释: `src-tauri/src/admin/mod.rs:1-10`
- 当前 release note 自相矛盾: `docs/release-notes-v2.0.0.md:3` 声称功能完全兑现, `docs/release-notes-v2.0.0.md:101-105` 又列出多个 stub

**推进要求**:

- 每个 stub 必须优先按 `v1.0.x` 旧版实现直接翻译, 保持旧接口返回结构和错误语义。
- 禁止保留“返回 success 但不实际工作”的行为。
- 只有确认旧逻辑在当前架构没有等价运行条件时, 才允许记录阻塞并暂缓, 不能自行设计新语义替代。

**验收**:

- 前端所有可点击入口不会因为缺少字段抛异常。
- 未实现功能不会显示为成功完成。

**当前状态**: 已完成。已移除 P0 范围内的假成功 stub, 改为按 `v1.0.3` 旧实现转译 provider 测试、模型探测、用量查询、兼容性矩阵、配置备份/导入、代理日志/统计和反馈提交。

## 建议推进顺序

1. `P0.0` 先处理所有假成功行为, 按 `v1.0.x` 旧逻辑直接翻译成真实实现。
2. `P0.1` 到 `P0.5` 按用户可见风险逐项恢复真实能力。
3. `P1.1` 到 `P1.4` 恢复 v1.0.3 的核心产品体验。
4. `P2.1` 到 `P2.4` 补协议兼容和边界场景。
