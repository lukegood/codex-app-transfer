# Issues

## 2026-05-12 Kimi Code `/responses/compact` token-limit failure

- Evidence: `反馈/0512-1/[反馈] (无标题) · fb-da4d0cc8.pdf` says Kimi research causes context explosion and that compact sends much more context than normal dialogue. The screenshot `反馈/0512-1/image.png` shows a new conversation doing a large research task with many web/code-search/tool steps, then failing compact with `requested: 336979`, limit `262144`, while the UI says `244k / 245k tokens used`.
- Full folder check: the local folder contains the email PDF, screenshot, three request bundles, `codex-config.redacted.toml`, and `proxy-config.redacted.json`. The email lists `versions.txt` and `proxy-tail-2026-05-12.log`, but those two files are not present in `反馈/0512-1/`.
- Main failure bundle: `反馈/0512-1/bundle-20260512-232148-61422-1778599308510.json` recorded `POST /responses/compact` routed to Kimi Code (`https://api.kimi.com/coding/v1/chat/completions`) and failed with upstream HTTP 400: `requested: 336979`, limit `262144`. The actual outbound request body was 854,476 bytes, and diagnostics truncated 592,332 bytes from the saved copy.
- Why one large research task became huge: the Responses-to-Chat conversion keeps `function_call_output` as Chat `tool` messages with full `content`. For research sessions, this means web/search/file outputs are retained as conversation content instead of being stored as bounded artifacts or compacted excerpts. The saved prefix already contains a 54,578-character system message, a 26,286-character user message, and at least 113,005 characters of complete tool-message content before truncation.
- Why compact grew after the threshold: `crates/adapters/src/responses/compact.rs` keeps the original `input` array, appends a large local compact prompt, and forwards the result through the generic chat conversion path. The compact path has no input budget or pruning step. The code comments reference Codex's `COMPACT_USER_MESSAGE_MAX_TOKENS`, but this project implements it as `COMPACT_MAX_OUTPUT_TOKENS = 20_000`, so the 20K budget limits summary output instead of limiting compact input.
- Why the UI shows about 244k instead of Kimi's advertised 256k: the model catalog sets Kimi's `context_window` to 258,400 and applies `effective_context_window_percent = 95`, so the effective visible budget is about 245,480 tokens. This matches the screenshot's `245k` limit and is intentional headroom for provider overhead, output tokens, and request wrapping.
- Root cause: this is a compact request-construction bug. Auto-compact is triggered near the effective threshold, but the compact request itself sends the whole already-large history plus extra summarization instructions and output allowance to Kimi. The correct fix is compact-specific input budgeting and evidence-aware pruning/summarization of tool outputs, not simply lowering the auto-compact trigger threshold.
- Implemented mitigation: long `function_call_output` payloads now go through a shared normalization layer before entering Chat `tool.content`. Raw payloads are saved in a sidecar SQLite store at `~/.codex-app-transfer/tool_artifacts.db`; model-visible history gets an artifact ID, tool call ID, original size metadata, path/URL hints, and bounded head/tail excerpts instead of full raw output. This applies to the OpenAI-compatible Responses path, `/responses/compact`, and Gemini native normalized-chat path.
- Implemented compact input budget: `/responses/compact` now applies a compact-only budget after Responses-to-Chat conversion. The final summarization prompt remains last, newest conversation blocks are preserved first, assistant tool-call groups stay attached to following tool outputs, and any omitted or shortened older content is represented by an explicit compact budget notice with counts and user-message excerpts.
- Validation: `cargo fmt --check`, `cargo test -p codex-app-transfer-adapters build_compact_chat_request_`, `cargo test -p codex-app-transfer-adapters artifact`, `cargo test -p codex-app-transfer-adapters large_function_call_output`, `cargo test -p codex-app-transfer-adapters large_tool_output`, `cargo test -p codex-app-transfer-adapters`, and `cargo test -p codex-app-transfer-registry` passed on 2026-05-13.
- Related but separate artifacts: the two Grok Web bundles in the same folder show 401 `No credentials presented`; they are unrelated to the Kimi compact failure.

## 2026-05-13 Anyrouter Claude sub-model routing and native web search

- Evidence: `~/.codex-app-transfer/logs/proxy-2026-05-13.log` shows Claude/Codex tasks can issue non-default inbound model names such as `gpt-5.4` and `gpt-5.4-mini`. In the same log around 16:26:41 they mapped to `claude-opus-4-7`; around 16:27:17 to 16:28:47 a `gpt-5.4` inbound request still reached upstream as `model:"claude-opus-4-7"` and received Anyrouter 429 `Service Unavailable`.
- Root cause: the Anyrouter integration cannot rely on the client always requesting the visible default model. Claude/Codex may spawn internal sub-model calls. Known OpenAI slot aliases were already mapped, but unknown future aliases or slash-route explicit model names could still bypass the intended default route.
- Implemented mitigation: Anyrouter preset now sets `requestOptions.proxy.force_default_model=true`, and resolver logic maps every inbound model name for that provider to `models.default`, including unknown model names and provider slash routes. A regression test preserves old slash-route behavior for providers that do not opt into force-default routing.
- Protocol follow-up: Anyrouter also enables Anthropic native hosted web search by mapping Responses `web_search` to Anthropic `web_search_20250305`, and response conversion now preserves Anthropic `server_tool_use` web search calls, `web_search_tool_result` URL annotations, and thinking signatures for `previous_response_id` continuation.
- Validation: `cargo fmt --all`, adapter request/response tests, full adapter tests, registry tests, targeted resolver tests, and full proxy tests passed on 2026-05-13. Full proxy tests require local loopback binding outside the default sandbox.

## 2026-05-13 Anyrouter forced tool-call and continuation

- Evidence: `~/.codex-app-transfer/logs/proxy-2026-05-13.log` showed the failed forced tool-call request already reached Anyrouter as `model:"claude-opus-4-7"` with the expected function tool and `tool_choice:{"type":"tool","name":"echo_probe"}`. The same final Anthropic Messages body also contained `thinking:{"type":"adaptive"}`.
- Root cause: Anthropic extended thinking is incompatible with forced `tool_choice.type=any/tool`; only `auto` and `none` are compatible. LiteLLM maps named function tool choice to Anthropic `tool_choice.type=tool` and drops `thinking` in tool-continuation conflict cases to preserve tool-round validity.
- Anyrouter A/B finding: forced `tool_choice.type=tool` plus adaptive thinking returned 520 in direct Anyrouter probing and 429 in the proxy streaming path. The same forced tool request without thinking returned 200 and produced `tool_use`. Tool-capable requests with adaptive thinking but no forced tool choice, or explicit `auto`, returned 200.
- Implemented mitigation: the Anthropic Messages request adapter removes `thinking` and `output_config.effort` when the final wire request contains forced `tool_choice.type=any/tool`, while preserving the forced `tool_choice` itself.
- End-to-end validation: a temporary `.tmp/anyrouter-e2e` probe started the current worktree proxy router, loaded the local Anyrouter provider with builtin healing, sent a forced `tool_choice` request using inbound `model=gpt-5.4-mini`, and received `function_call(name=echo_probe, arguments={"value":"ok"})`. The follow-up request used `previous_response_id` plus `function_call_output` and completed with `output_text=done` and `response.completed.status=completed`.
