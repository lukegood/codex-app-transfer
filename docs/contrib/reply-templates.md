# 固定回复模板

## PR 作者整改回复

适用场景:对外部作者提交的 PR 做过深入 review、补充整改并合并后,需要在 PR discussion 中说明分析结果、解决方式和感谢。

回复风格:

- 先感谢作者,并明确认可 PR 中正确的核心方向。
- 用事实说明本地或 CI 验证过的关键判断,避免只给主观结论。
- 分析部分使用编号列表,每一点只讲一个边界问题。
- 解决方式部分用一段话串起来,说明保留了作者方案中的正确方向,以及补齐了哪些工程边界。
- 验证命令用代码块列出,让作者知道合并前跑过哪些检查。
- 结尾说明 PR 当前状态,再次感谢作者贡献。

固定模板:

```markdown
@<author> 感谢这个 PR。你指出的 <核心兼容方向或核心修复点> 是对的。我本地也验证了:<关键验证事实 1>；<关键验证事实 2>。

合并前我重点分析了几个边界问题，并在这个 PR 里一起修正了：

1. <问题 1:说明为什么会受影响,重点区分表象和根因。>
2. <问题 2:说明生命周期、恢复、清理、兼容性或用户已有配置等边界。>
3. <问题 3:说明路由、映射、能力声明或数据来源需要和真实配置保持一致。>
4. <问题 4:说明内部标记、兼容标记或特殊输入不能泄漏到外部边界。>
5. <问题 5:说明被移出或延后的内容,以及为什么不应混在本次 PR 中。>

最终处理方式是：保留你的 <被认可的核心方向>，同时 <修正 1>；<修正 2>；<修正 3>；并 <移出或延后项>。

我已经补了对应回归测试，并验证了以下命令：

```bash
<validation command 1>
<validation command 2>
<validation command 3>
```

这个 PR 已经 <merged / ready to merge / 仍需等待的状态>。再次感谢你补上 <具体价值点>。
```

使用要求:

- `<author>` 使用 PR 作者的 GitHub handle。
- `<核心兼容方向或核心修复点>` 要具体到作者真正提供的贡献,不要泛泛写“修复问题”。
- 分析列表可以少于 5 项,但必须覆盖 review 中真正影响合并判断的问题。
- 如果 PR 尚未合并,最后一句改为当前真实状态,例如“这个 PR 已经通过 CI,我会在确认发布窗口后合并”。
- 如果本次没有新增测试,不要写“补了对应回归测试”,改为真实验证方式。
