## ❌ 缺失项 (Python 有,Rust 无)

### 1. Response ID 编码/解码与 Deployment Affinity

**影响严重度**: 🟡 设计差异(非缺失,改重评)—— v2.0.7 已通过"切 provider 后用户重启 Codex CLI"硬切换规避

- **Python**: `backend/response_id_codec.py:1-199` 
  - `encode_response_id(provider, model, request_id)` — 将 provider/model/request_id 编码为 `resp_{base64(litellm:custom_llm_provider:...)}` 格式
  - `decode_response_id(response_id)` — 解码并提取 `custom_llm_provider`、`model_id`、`response_id`
  - `encode_container_id()` / `decode_container_id()` / `encode_content_item_id()` — 其他 ID 类型编解码
  
- **Python**: `backend/deployment_affinity.py:L57-132`
  - `check_deployment_affinity(body, active_provider, available_providers)` — 校验 `previous_response_id` 中编码的 provider 是否与当前激活 provider 匹配,不匹配时返回建议 provider
  - `resolve_provider_for_previous_response(previous_response_id, available_providers)` — 根据编码的 provider 查找最佳匹配提供商

- **Rust**: 搜过 `crates/` 和 `src-tauri/`,**无等价实现**
  - Session cache 能存储历史消息,但缺少 affinity 校验
  - Response ID 生成为简单 `resp_{random}` 格式,无编码信息
  - 无法在多 provider 场景下维持会话路由粘性 (sticky routing)

- **用户侧表现**: 
  - 单 provider 场景:无感(历史消息能通过 `previous_response_id` 恢复)
  - 多 provider 场景:用户在 provider A 问题,得到 `resp_xxx`,然后切到 provider B 继续追问,无法自动路由回 A,可能导致会话连贯性破裂

- **建议**: 维持现状或低优先补齐
  - v2.0.7 引导用户切 provider 后**重启终端**(参见 `frontend/js/i18n.js:485` "After switching providers, you must close and reopen the terminal"),Codex CLI 重启 = 上下文清空 = 不会跨 provider 带 `previous_response_id`,从产品设计上规避了此问题
  - 若未来要支持"不重启 Codex CLI 也能切 provider 续轮",再补齐 codec + affinity 校验

### 2. Proxy 服务缺少 `/health` 和 `/status` 端点

**影响严重度**: 🟡 中等 —— Codex CLI 可能无法判断代理健康状态

- **Python**: `backend/proxy.py:L933-936`
  ```python
  @app.get("/health")
  @app.get("/status")
  async def health():
      return {"status": "ok", "stats": stats.to_dict()}
  ```

- **Rust**: `crates/proxy/src/server.rs:L43` `.fallback(any(forward_handler))` —— 所有路径直接转发到上游,不拦截 `/health` 和 `/status`

- **用户侧表现**: 
  - Codex CLI 或其他客户端如果在请求模型前先探测代理健康状态,会被误转发到上游,可能产生意外的 API 调用
  - 无法直接查看本地代理统计(request count, success rate 等)

- **建议**: 在 proxy server 加两个路由:
  ```rust
  .route("/health", get(health_handler))
  .route("/status", get(health_handler))
  ```
  返回类似 `{"status": "ok", "stats": {...}}`

## 总结(经 2026-05-07 反向核实修正后)

**真正的 ❌ 缺失项: 1 个**

1. ❌ **代理服务 `/health` 和 `/status` 端点** — Python 在 proxy port 暴露 `{"status":"ok","stats":{...}}`,Rust fallback 把 `/health` 当成 chat/completions 透传到上游。客户端 / 监控脚本探测代理健康会误打上游。

**🔄 设计差异(非缺失): 1 个**

1. 🔄 **Response ID 编码 / Deployment Affinity** — Python 把 provider 编进 `resp_xxx`(base64 of `litellm:custom_llm_provider:X;model_id:Y;response_id:Z`)用于 multi-provider sticky routing。v2.0.7 走 "切 provider 必重启 Codex CLI" 的硬切换,Codex CLI 重启即清 `previous_response_id`,产品上规避。**不算漏功能**,等未来要支持"不重启换 provider 续轮"再做。

**⚠️ 部分对齐项: 4 个**(已剔除误判的 #5 日志 clear)

1. ⚠️ **Provider workaround dispatch 不完整** — Python 覆盖 anthropic/mistral/gemini/groq/together;v2.0.7 只接 deepseek/kimi/mimo,对未来扩展新 provider 缺统一入口。**当前用户场景下不影响**,接到新 provider 再补。
2. ⚠️ **requestOptions 深度合并** — Rust 单层读,Python 递归 merge。仅在嵌套配置(`extra_body.xxx`)下有差。
3. ⚠️ **extraHeaders `{apiKey}` 模板替换** — Rust 直传字面量。低概率影响,但**修起来一行代码**,值得顺手补。
4. ⚠️ **模型 `supports1m` 完整判断** — Rust 只识别 `[1m]` 后缀,缺 `modelCapabilities.supports1m` 查询 + 已知 1M 族默认值。仅影响 UI 标记。
